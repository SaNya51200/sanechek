## 2. Теоретические сведения

### Основные концепции (Часть 1: React + Vite)
- **Компонентно-ориентированный подход:** Разделение пользовательского интерфейса на независимые, изолированные и переиспользуемые блоки (компоненты).
- **Хуки (Hooks):** Специальные функции, позволяющие использовать состояние React без написания классов. Основным хуком для управления локальным состоянием является `useState`.
- **Иммутабельность состояния:** Принцип неизменяемости данных в React. Обновление состояния должно происходить путем передачи новой копии объекта или массива через функцию-сеттер (например, `setTaskList`), что гарантирует корректный запуск цикла рендеринга.
- **TypeScript в React:** Позволяет строго описывать типы входных параметров компонентов (`props`) и объектов состояния, снижая риск ошибок времени выполнения.

### Основные концепции (Часть 2: Next.js)
- **App Router:** Новая файловая система маршрутизации, где пути определяются структурой вложенных папок в директории `app`.
- **Server Components:** Компоненты, которые рендерятся на сервере по умолчанию. Это позволяет уменьшить объем отправляемого клиенту JavaScript-кода и оптимизирует производительность приложения.
- **SSG (Static Site Generation):** Процесс предварительного рендеринга статических HTML-страниц на этапе сборки приложения (`npm run build`). Для динамических путей в Next.js используется функция `generateStaticParams`.

### Часть 1: Менеджер задач (todo-app)
Было создано модульное React-приложение со следующей структурой компонентов:
1. `types.ts` — объявление интерфейса данных задачи `ITask` (`uid`, `text`, `isCompleted`).
2. `TaskInput.tsx` — поле ввода для создания новых задач.
3. `TaskItem.tsx` — строка отдельной задачи с чекбоксом выполнения и кнопкой удаления.
4. `TaskList.tsx` — контейнер списка задач с обработкой пустого состояния.
5. `TaskStats.tsx` — панель аналитики и прогресс-бар выполнения задач.

#### Ключевой фрагмент управления состоянием (App.tsx):
```tsx
// Добавление задачи
const handleAddTask = (text: string) => {
  const newTask: ITask = {
    uid: crypto.randomUUID(),
    text,
    isCompleted: false,
  };
  setTaskList([...taskList, newTask]);
};

// Изменение статуса выполнения
const handleToggleStatus = (uid: string) => {
  setTaskList(
    taskList.map((task) =>
      task.uid === uid ? { ...task, isCompleted: !task.isCompleted } : task
    )
  );
};
```

## 4. Контрольные вопросы

### Часть 1: React & Vite
1. **Принцип работы useState:**  
   Хук `useState` инициализирует состояние компонента. При вызове возвращаемой им функции-модификатора (например, `setTaskList`), React планирует повторный рендер компонента с новым значением состояния, обновляя виртуальное дерево DOM (Virtual DOM).
2. **Важность неизменяемого состояния в React:**  
   React сравнивает ссылки на объекты в состоянии по алгоритму поверхностного сравнения (Shallow Comparison). Если изменить свойства объекта или добавить элемент в массив напрямую (например, через `push`), ссылка останется прежней, и React не инициирует рендер интерфейса.
3. **Метод для удаления задач из списка:**  
   Используется метод `filter()`, который создает новый отфильтрованный массив на основе заданного предиката. Это позволяет следовать правилу иммутабельности.
4. **Преимущества статической типизации в React:**  
   TypeScript помогает избежать ошибок несоответствия типов пропсов при передаче данных в дочерние компоненты и облегчает документирование структуры интерфейсов и API.

### Часть 2: Next.js
5. **Понятие Static Site Generation (SSG):**  
   Это метод, при котором HTML-код страниц генерируется на этапе компиляции приложения. Это ускоряет загрузку сайта, повышает производительность сервера и упрощает поисковую индексацию (SEO).
6. **Принцип работы файловой маршрутизации:**  
   Каждая папка внутри папки `/app`, содержащая файл `page.tsx`, автоматически становится доступным URL-адресом. Глубина вложенности папок определяет структуру URL.
7. **Роль функции `generateStaticParams`:**  
   Функция указывает Next.js, какие именно динамические пути (например, `/blog/[slug]`) должны быть скомпилированы в статические HTML-страницы на этапе сборки.
8. **Различия сборки и режима разработки:**  
   Режим разработки (`npm run dev`) компилирует страницы по мере необходимости и поддерживает горячую замену модулей (HMR). Сборка (`npm run build`) оптимизирует код, минифицирует файлы и генерирует статический контент для продакшн-запуска.
