## 2. Теоретические сведения

### ETL-архитектура
- **Extract (Извлечение):** Чтение сырых данных из различных источников (файлы, сторонние API, операционные базы данных).
- **Transform (Преобразование):** Очистка данных (удаление дубликатов, заполнение пропущенных значений, фильтрация аномалий) и приведение их к единому стандарту, а также расчет аналитических агрегатов.
- **Load (Загрузка):** Сохранение очищенных данных в аналитическое хранилище (DWH) или реляционную БД.
- **Визуализация:** Построение графиков для выявления трендов и представления результатов бизнеса.

### Apache Kafka
- **Event Streaming:** Платформа распределенного стриминга событий, позволяющая обрабатывать потоки данных в реальном времени.
- **Producer (Производитель):** Приложение, генерирующее события и отправляющее их в определенный топик Kafka.
- **Consumer (Потребитель):** Приложение, подписывающееся на топики и считывающее поступающие сообщения для их последующей обработки.
- **Consumer Group (Группа потребителей):** Множество потребителей, совместно считывающих данные из топика. Разделы топика распределяются между членами группы, обеспечивая масштабируемость и отказоустойчивость.
- **Временные окна (Windowing):** Подход в обработке потоков, при котором данные группируются по временным интервалам (например, каждые 30 секунд) для выполнения скользящего или интервального анализа метрик.

## 4. Контрольные вопросы

1. **Задачи стадий ETL-процесса:**  
   `Extract` получает данные из сырых источников. `Transform` проверяет типы, фильтрует невалидные записи, дедуплицирует строки и вычисляет агрегированные метрики. `Load` записывает чистые данные в СУБД/хранилище для конечных бизнес-пользователей.
2. **Как работает скользящее временное окно (Sliding Window) в потоковой обработке:**  
   В отличие от интервальных (Tumbling) окон, скользящее окно сдвигается с течением времени на фиксированный шаг (например, окно 10 минут сдвигается каждые 10 секунд). События попадают в окно в зависимости от их меток времени, позволяя получать непрерывные метрики.
3. **Назначение Zookeeper / Kraft в Apache Kafka:**  
   Используются для координации распределенного кластера Kafka: управление метаданными, выбор контроллера (leader election), отслеживание состояния брокеров и топиков. KRaft — это современный встроенный протокол консенсуса, исключающий необходимость запуска внешнего Zookeeper.
4. **Механизм масштабирования чтения через Consumer Group:**  
   Топик Kafka состоит из нескольких разделов (partitions). Внутри одной Consumer Group каждый раздел читается только одним потребителем. Это позволяет распараллелить обработку данных: при добавлении новых консьюмеров происходит ребалансировка разделов.
