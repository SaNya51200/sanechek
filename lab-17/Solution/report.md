## 2. Облачный деплой приложения

### Подготовка frontend
Frontend-приложение на базе Next.js расположено в каталоге `tselenko-labs/lab-17/Solution/fullstack-app/frontend`.

Перед деплоем в облако контейнер был успешно собран и проверен локально:
```bash

docker run -d --name tselenko_api_test -p 18000:8000 tselenko-backend-api
curl http://localhost:18000/health
```

- **Backend** развернут в Yandex Cloud через службу Serverless Containers с автоматическим масштабированием.
- Для связи компонентов на Vercel прописана переменная `NEXT_PUBLIC_API_URL`, указывающая на адрес бэкенд-контейнера в Yandex Cloud.

## 3. Настройка CI/CD пайплайна

Были созданы три workflow файла автоматизации процессов в каталоге `.github/workflows`:

1. **`deploy.yml` (Production деплой):**  
   Срабатывает при слиянии изменений в ветку `main`.
   - Задача `deploy-frontend`: собирает Next.js и выкладывает проект в Vercel через Vercel CLI.
   - Задача `deploy-backend`: авторизуется в Yandex Container Registry, компилирует Docker-образ, пушит его в реестр и создает новую ревизию контейнера в Yandex Serverless.

2. **`pr-checks.yml` (Тесты на Pull Request):**  
   Запускается при создании PR. Проводит установку зависимостей, сборку фронтенда и компиляцию Python-кода бэкенда для защиты основной ветки от синтаксических ошибок.

3. **`terraform.yml` (IaC валидация):**  
   Выполняет автоматическую проверку инфраструктурного кода: форматирование (`terraform fmt`), инициализацию (`terraform init`) и синтаксическую валидацию (`terraform validate`).

#### Фрагмент workflow проверки пул-реквестов (`pr-checks.yml`):
```yaml
name: Pull Request Checks

on:
  pull_request:
  workflow_dispatch:

jobs:
  frontend:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: tselenko-labs/lab-17/Solution/fullstack-app/frontend
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
          cache-dependency-path: tselenko-labs/lab-17/Solution/fullstack-app/frontend/package-lock.json
      - run: npm ci
      - run: npm run build
  # ...
```

## 4. Контрольные вопросы

### Часть 1 (Деплой)
1. **Преимущества многослойной сборки Docker:**  
   Позволяет разделить сборку на этапы (например, компиляция и выполнение). За счет кэширования слоев ускоряется сборка, а из итогового образа вырезаются компиляторы и лишние инструменты, уменьшая размер контейнера и повышая безопасность.
2. **Обоснование выбора Vercel и Yandex Cloud:**  
   Vercel — специализированный хостинг с мгновенным Edge-рендерингом для Next.js и автоматическим деплоем. Yandex Cloud подходит для хостинга бэкенда благодаря низкому пингу в РФ, поддержке Serverless-контейнеров, управляемых баз данных и Yandex Container Registry.
3. **Решение проблемы CORS:**  
   В FastAPI-приложении подключен модуль `CORSMiddleware`. В параметре `allow_origins` передается список разрешенных хостов (включая домен фронтенда с Vercel), что разрешает браузеру выполнять AJAX-запросы.
4. **Секретные и публичные переменные:**  
   Секретными должны быть авторизационные ключи (JSON ключи аккаунтов, токены, пароли к БД). Они хранятся в GitHub Secrets и секретах облака. Публичными могут быть адреса внешних API (например, `NEXT_PUBLIC_API_URL`), которые компилируются прямо в клиентский JS-код.
5. **Понятие холодного старта:**  
   Задержка ответа при первом за долгое время запросе к Serverless контейнеру. Возникает из-за необходимости облака поднять экземпляр контейнера, инициализировать ОС и запустить рантайм приложения.

### Часть 2 (CI/CD)
6. **Плюсы CI/CD перед ручным деплоем:**  
   Автоматизация исключает человеческий фактор (забытые тесты, неправильные переменные), гарантирует повторяемость сборки и значительно ускоряет поставку фич в продакшн.
7. **Роль GitHub Actions Secrets:**  
   Предотвращают утечку секретов в публичный репозиторий. Значения шифруются и подставляются в шаги сборщика только во время выполнения пайплайна, не отображаясь в логах Actions.
8. **Назначение проверок на Pull Request:**  
   Позволяют автоматически провалидировать изменения (тесты, линтеры, сборка) до их вливания в ветку `main`, снижая вероятность сломать рабочее окружение.
9. **Зачем проверять Terraform в CI/CD:**  
   Инфраструктура описывается кодом (IaC). Проверки `fmt` и `validate` гарантируют отсутствие синтаксических ошибок в файлах конфигурации облака еще до запуска процесса развертывания ресурсов.
10. **Поведение пайплайна при ошибке:**  
    Если один из шагов сборки (например, тесты) падает с ненулевым кодом выхода, пайплайн прерывается, блокируя последующий шаг деплоя, что предотвращает выкат нерабочего кода в прод.
